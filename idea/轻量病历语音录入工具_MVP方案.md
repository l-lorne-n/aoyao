# 轻量病历语音录入工具 MVP 方案

> 版本：V1.0
> 日期：2026-06-26
> 当前定位：只解决医生在填写纸质/电子记录时的语音转文字问题，不建设完整诊所管理系统。

\---

## 1\. 项目背景

目前诊所主要通过纸质表格记录患者信息，医生需要手动填写“主诉”“辨证”“内调方案”“回访情况”等自由文本，书写负担较大。

本项目计划开发一个轻量语音录入工具，让医生点击对应文本框旁的录音按钮，通过口述直接生成文字，再由医生检查、修改和确认。

医生使用的电脑为普通轻薄本，因此不在本地运行语音识别模型。录音由电脑采集，语音识别在云端完成，电脑只负责页面展示和文字编辑。

\---

## 2\. 当前需求范围

### 2.1 第一版需要实现

* 在“主诉”字段旁提供语音输入按钮；
* 在“回访情况”字段旁提供语音输入按钮；
* 后续可为“辨证”和“内调方案”增加同样的语音按钮；
* 点击开始录音，口述完成后点击停止；
* 将录音发送到云端语音识别接口；
* 返回识别文字并显示在编辑框中；
* 支持医生手动修改；
* 支持将识别结果追加到原内容或替换原内容；
* 识别完成后删除临时音频。

### 2.2 第一版明确不做

* 不接入卫健委、医保或其他监管平台；
* 不接入收费、库存、处方流转系统；
* 不录制完整医患对话；
* 不自动诊断；
* 不自动生成治疗方案；
* 不使用大语言模型总结或改写；
* 不在轻薄本上部署本地语音模型；
* 不建设复杂的多医生权限系统。

\---

## 3\. 推荐交互流程

```text
医生点击“开始录音”
        ↓
医生口述主诉或回访内容
        ↓
医生点击“结束并识别”
        ↓
系统上传本次短音频
        ↓
云端语音识别返回文字
        ↓
医生检查和修改
        ↓
选择“追加”“替换”或“取消”
        ↓
保存最终文字
```

页面示例：

```text
主诉
┌────────────────────────────────────┐
│ 患者近一个月睡眠欠佳，入睡困难…… │
└────────────────────────────────────┘
\[开始录音] \[结束并识别] \[重新录音]

识别结果：
患者近一个月睡眠欠佳，入睡困难，夜间易醒。

\[追加到原内容] \[替换原内容] \[取消]
```

每个文本字段应使用独立录音按钮，避免系统不知道识别结果应填入哪个位置。

\---

## 4\. 语音识别接口选择

### 4.1 首选：腾讯云一句话识别

第一版推荐：

* 产品：腾讯云“一句话识别”；
* 识别引擎：`16k\_zh\_medical`；
* 使用方式：录完一段后上传，等待返回完整文字；
* 适合字段：主诉、回访情况、辨证、内调方案；
* 单段限制：音频不超过 60 秒。

选择理由：

1. 明确提供中文医疗识别引擎；
2. 一句话识别支持 `16k\_zh\_medical`；
3. 适合一次几十秒的病历口述；
4. 接口为普通 HTTP 请求，第一版开发难度低；
5. 支持热词和临时热词能力；
6. 不需要在本地电脑运行模型；
7. 识别失败时不会产生接口费用。

#### 为什么暂时不做实时识别

实时识别能够边说边显示，但需要维护 WebSocket 连接、处理临时结果覆盖、断线重连和文字重复等问题。

当前使用场景是“点一下，说一段，结束后填入文本框”，因此录制完成后一次识别更加稳定，开发也更简单。

若以后医生经常需要连续口述超过 60 秒，再升级到：

* 腾讯云实时语音识别 WebSocket；
* 同样使用 `16k\_zh\_medical` 医疗引擎。

### 4.2 备选：科大讯飞实时语音转写

若腾讯云对医生口音、中医术语或固定表达的识别效果不理想，可以测试科大讯飞。

备选配置：

* 科大讯飞实时语音转写；
* 医疗领域参数：`medical`；
* 配置个性化热词；
* 通过 WebSocket 上传音频。

讯飞适合作为第二家供应商进行效果对比，而不是第一版同时接入两套接口。

## 5\. 热词设计

医学语音识别效果不仅取决于供应商，也取决于热词配置。

建议根据医生实际使用习惯建立热词表，第一批可包含：

```text
口干口渴
口淡
口苦
怕冷
怕热
嗜睡
失眠多梦
纳呆
大便干硬
大便稀溏
尿急尿频
夜尿
淋漓不尽
舌脉象
舌质淡红
舌质暗红
苔薄白
脉弦
脉细
辨证
内调方案
健脾祛湿
疏肝解郁
滋阴清热
```

后续可记录“经常被识别错的词”，每周更新一次热词表。

注意：热词只能提高相关词出现时的识别概率，不能代替医生最终核对。

## 6\. 技术实现方案

### 6.1 推荐架构

```text
浏览器页面
  ├── 主诉输入框
  ├── 回访输入框
  └── 录音按钮
          ↓
Python 后端
  ├── 接收录音
  ├── 必要时转换音频格式
  ├── 调用腾讯云 ASR
  ├── 返回识别文字
  └── 删除临时音频
          ↓
腾讯云语音识别
```

### 6.2 推荐技术栈

第一版可以使用：

* 页面：Streamlit；
* 后端：Python；
* 录音：浏览器 MediaRecorder 或 Streamlit 录音组件；
* 语音接口：腾讯云一句话识别；
* 音频转换：FFmpeg；
* 配置管理：`.env` 环境变量；
* 运行方式：轻薄本本地启动，浏览器访问。

如果后续需要更正式的界面，可将前端换成 React/Next.js，后端换成 FastAPI，但第一版没有必要。

### 6.3 统一接口设计

业务页面不要直接耦合腾讯云 SDK，建议封装统一函数：

```python
def transcribe\_audio(audio\_path: str) -> str:
    # 将本地短音频发送到语音识别服务并返回文字
    ...
```

后续若更换为讯飞，只需要替换函数内部实现，不需要重写页面。

可以进一步定义供应商适配层：

```text
ASRProvider
├── TencentASRProvider
└── XunfeiASRProvider
```

## 7\. 音频要求

为了提高识别稳定性，推荐统一使用：

* 采样率：16 kHz；
* 位深：16 bit；
* 声道：单声道；
* 优先格式：WAV 或 PCM；
* 单次时长：建议 10～50 秒；
* 最长不超过 60 秒。

浏览器录音可能生成 WebM/Opus。后端可以先通过 FFmpeg 转换成 16 kHz 单声道 WAV，再调用接口。

示例命令：

```bash
ffmpeg -i input.webm -ar 16000 -ac 1 -sample\_fmt s16 output.wav
```

腾讯云一句话识别本身也支持多种音频格式，但统一格式有利于排查问题和比较不同供应商的效果。

## 8\. 隐私与安全要求

虽然当前系统很简单，但录音和识别文字可能包含患者健康信息，需要进行基本保护。

### 必须做到

* 腾讯云密钥只能放在 Python 后端；
* 不得将 `SecretID`、`SecretKey` 写进网页前端；
* `.env` 文件不得提交到 GitHub；
* 临时录音使用随机文件名；
* 识别完成后立即删除临时音频；
* 默认不长期保存患者录音；
* 程序日志不记录完整主诉或回访内容；
* 接口报错时不在页面显示密钥或完整请求信息；
* 正式使用前关闭调试模式。

### 推荐做到

* 电脑设置开机密码；
* 病历数据定期备份；
* 使用 HTTPS/WSS 与云服务通信；
* 使用最小权限的云端 API 子账号；
* 定期更换 API 密钥；
* 真实使用时避免把姓名、电话等身份信息读进语音中。

## 9\. 费用估算

### 腾讯云

腾讯云当前公开计费页显示：

* 一句话识别：3 万次资源包，90 元，有效期 1 年；
* 实时语音识别：30 小时资源包，90 元，有效期 1 年；
* 开通服务后会发放免费资源包，具体剩余额度以控制台为准。

按当前项目的使用方式，优先使用“一句话识别”。

假设：

* 每天 20 位患者；
* 每位患者录入 2 段；
* 每月工作 25 天；

则每月约调用：

```text
20 × 2 × 25 = 1000 次
```

3 万次资源包理论上可覆盖约 30 个月的上述使用量，但资源包有效期为 1 年，因此实际应按一年有效期计算。即使达到上述使用量，接口成本仍然很低。

### 科大讯飞

讯飞官方文档显示，新用户根据认证情况，实时语音转写可获得最多 50 小时免费时长，有效期一年；正式价格需要登录控制台查看。

### 费用结论

当前项目不需要因价格选择识别效果较差的方案。主要决策标准应为：

1. 医学术语识别准确率；
2. 否定词识别准确率；
3. 医生口音适配；
4. 返回速度；
5. 接入稳定性。

## 10\. 测试方案

在正式接入病历页面前，先制作一个独立测试页：

```text
\[开始录音]
\[结束并识别]

识别结果：
\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_

接口耗时：
\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_
```

让医生录制 20～30 段匿名测试语音，内容应覆盖：

* 主诉；
* 第一次回访；
* 多次回访；
* 中医术语；
* 较快语速；
* 自然停顿；
* 带口音普通话；
* 否定表达；
* 数字和时间。

重点检查以下内容：

### 10.1 否定词

```text
无头晕
没有胸闷
不伴发热
未见明显改善
```

否定词一旦漏识别，可能改变原意，应作为最高优先级检查项。

### 10.2 医学术语

记录每个被识别错误的词，并判断：

* 是否可以加入热词；
* 是否需要替换词；
* 是否属于麦克风或发音问题；
* 是否需要更换供应商。

### 10.3 数字

重点测试：

```text
三天
两周
每天两次
夜尿三次
血压一百二十八毫米汞柱
```

### 10.4 测试指标

建议记录：

|指标|目标|
|-|-|
|普通文字可直接采用率|≥ 90%|
|关键医学术语正确率|≥ 95%|
|否定词错误|0 个|
|30 秒音频返回时间|尽量 ≤ 5 秒|
|接口调用成功率|≥ 99%|
|录音后临时文件残留|0 个|

这里的目标用于项目验收，不代表供应商官方承诺。

## 11\. 异常处理

系统至少需要处理以下情况：

### 录音权限被拒绝

提示：

```text
无法访问麦克风，请在浏览器设置中允许本页面使用麦克风。
```

### 网络断开

提示：

```text
网络连接失败，本次录音尚未写入文本框。请检查网络后重新识别。
```

### 接口超时

保留本次录音，允许医生点击“重新识别”，不要立即丢失音频。

### 音频过长

在录音接近 55 秒时提醒，并在 60 秒时自动停止，避免超过一句话识别限制。

### 返回空文本

提示医生重新录音，并检查麦克风是否正常。

### 识别结果错误

必须允许医生直接编辑，不能强制采用识别结果。

## 12\. 开发阶段安排

### 阶段一：接口验证

目标：

* 完成录音；
* 调用腾讯云一句话识别；
* 使用 `16k\_zh\_medical`；
* 显示识别结果；
* 测量耗时；
* 不接入病历页面。

验收标准：

* 能连续完成 20 次录音识别；
* 无程序崩溃；
* 临时音频可自动删除。

### 阶段二：接入主诉与回访

目标：

* 给“主诉”和“回访情况”分别添加录音按钮；
* 支持追加、替换和取消；
* 支持重新录音；
* 保留医生手动编辑能力。

### 阶段三：热词优化

目标：

* 收集高频错词；
* 建立初始热词表；
* 对相同测试音频进行优化前后对比。

### 阶段四：备选接口测试

只有腾讯云效果不满足要求时，再接入讯飞并进行 A/B 测试。

## 13\. 第一版验收标准

第一版完成时，应满足：

* \[ ] 普通轻薄本可流畅运行；
* \[ ] 不需要独立显卡；
* \[ ] 可以录制 60 秒以内语音；
* \[ ] 可以调用中文医疗识别引擎；
* \[ ] 主诉和回访字段拥有独立录音按钮；
* \[ ] 识别结果不会自动覆盖原文字；
* \[ ] 支持追加、替换和取消；
* \[ ] 医生可以手动修改；
* \[ ] 接口密钥不暴露在前端；
* \[ ] 临时音频识别后自动删除；
* \[ ] 网络或接口失败时有明确提示；
* \[ ] 可配置医学热词；
* \[ ] 不使用大语言模型；
* \[ ] 不自动生成诊断和治疗建议。

## 14\. 最终推荐方案

当前项目建议采用：

```text
Streamlit 页面
+ Python 后端
+ 浏览器短语音录制
+ 腾讯云一句话识别
+ 16k\_zh\_medical 医疗引擎
+ 医学热词表
```

第一版先完成一个独立的“录音—识别—显示文字”测试页面。确认医生真实口述的准确率可以接受后，再把语音按钮接入“主诉”和“回访情况”文本框。

暂时不接入大语言模型，不做自动总结，也不做实时整段医患对话识别。这样能以最低开发成本验证最核心的需求，并且普通轻薄本完全可以承担客户端工作。
